DESIGN DECISIONS TO REVIEW

1. Availability overlap: BEFORE-INSERT trigger plus inline RPC pre-check (belt and suspenders). Trigger takes FOR UPDATE lock on suppliers row, checks for overlapping non-released non-expired blocks via GiST, raises P0001 availability_conflict. v1 assumes concurrent_event_limit=1.

Q1: correct call? Is there a hole? Does the FOR UPDATE on suppliers leak a serialisability gap when trigger fires on updates vs inserts, or when two different suppliers are touched simultaneously?

2. accept_quote_tx(p_quote_id, p_organizer_id, p_soft_hold_minutes default 2880), SECURITY DEFINER, GRANT to service_role only. Lock order across every RPC: quotes, then rfqs, then events, then suppliers. Body: lock quote and rfq, lock event and check organizer, insert booking awaiting_supplier, insert soft_hold availability_block (trigger enforces), update statuses, reject siblings.

Q2a: lock order deadlock-free across two organizers accepting different suppliers quotes for the same event? Quotes differ, events same, suppliers differ. They should serialize on the events row lock. Is that right or am I missing a cycle?

Q2b: SECURITY DEFINER + REVOKE ALL FROM public + GRANT EXECUTE TO service_role sufficient, or also set role inside function body for defense-in-depth?

Q2c: LOCK_TIMEOUT not set anywhere. If two 8-way concurrent Vitest tests hit the same event, will any session wait indefinitely? Should the RPC set local lock_timeout = 5s at the top?

3. upsert_quote_revision_tx SECURITY DEFINER service_role only. SELECT quote FOR UPDATE; insert quote if missing; SELECT max(version); insert revision with version=max+1, retry once on unique-violation 23505; update current_revision_id, status, sent_at.

Q3: does FOR UPDATE on quotes actually serialize the version race, or do I still need the retry? If the lock serializes fully, the retry is dead code. Is dropping the retry preferable for simplicity?

4. Snapshot canonicalization. canonicalize sorts keys lexicographically (UTF-16), strips undefined, preserves null and array order, NFC-normalizes strings. Money is integer halalas. Multipliers stringified (e.g. "1.15"). engine_version="1.0.0". inputs_digest over a stable input set. sha256 via node:crypto.

Q4a: RFC-8785 (JCS) has subtle rules about number serialization. If I restrict persisted numbers to integers only, am I safe from V8 JSON.stringify drift across Node versions? Any pitfall?

Q4b: security issue with exposing content_hash in snapshot_jsonb to organizer RLS?

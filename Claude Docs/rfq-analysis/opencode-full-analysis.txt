**Architecture**
- Quote comparison/print/export:
  - Good: `quotes/page.tsx`, `print/page.tsx`, and `export.csv/route.ts` all share `loadQuotesComparison()`, so auth and data math are centralized.
  - Weakness: localization is not centralized. The loader returns raw slugs/statuses, then page/print/CSV each invent their own formatting.
  - Weakness: the detail page `[quoteId]/page.tsx` is a separate presentation stack and reintroduces hardcoded English.
  - Weakness: server actions in `quotes/actions.ts` own user-facing copy and notification payload shaping, which makes i18n regressions easy.

- Admin RFQ monitor:
  - Good: list/detail already use `segmentNameFor`, `cityNameFor`, `categoryName`, and `formatDate` helpers.
  - Weakness: status translation is incomplete (`pending`, booking states).
  - Weakness: requirements values are not modeled as locale-aware domain labels.
  - Weakness: search operates on raw storage slugs, not localized search terms.

- i18n infrastructure:
  - Good: `src/i18n/request.ts` centralizes locale and message loading.
  - Good: domain helpers are pure locale-in/value-out.
  - Weakness: Zod locale registration is global and race-prone.

  - Weakness: there is no shared helper tier for money, generic numbers, lists, enum labels, or notification Intl locale tags.

- Notification templates:

  - Weakness: locale-tag decisions are duplicated inline.
